# 设计文档

## 1. 系统概述

本项目实现了一个基于 Python Socket 和 Tkinter 的局域网即时聊天系统，支持：

- 用户注册与登录
- 私聊与群聊
- 群组创建、加入、退出、加人
- 在线/离线状态同步
- 历史消息同步
- 文件分块传输与下载
- 群聊中的 `@AI` 智能问答
- 内容审核与敏感词拦截
- 客户端断线后的自动重连与自动重新登录

系统采用典型的 C/S（Client/Server）架构：客户端负责图形界面和交互，服务端负责连接管理、消息路由、状态同步、历史持久化、AI 调用与文件缓存。

## 2. 系统架构

系统架构图见同目录文件：

- `系统架构图.pdf`

整体可以概括为四层：

1. 客户端表示层  
   Tkinter GUI，负责登录、联系人/群组列表、聊天渲染、输入区、文件卡片和状态栏。
2. 通信协议层  
   使用 TCP 长连接，并在其上定义“4 字节长度头 + UTF-8 JSON 消息体”的自定义帧协议。
3. 服务端业务层  
   负责注册登录、消息路由、群组管理、心跳检测、文件传输、AI 调用和内容审核。
4. 数据持久化与外部能力层  
   使用 JSON 文件保存用户、群组和历史消息，使用 `data/files/` 存放文件内容，使用 GLM API 提供 AI 问答能力。

## 3. 协议定义

### 3.1 帧格式

每一条业务消息在网络上传输时，统一封装为：

```text
[4 字节长度头][UTF-8 JSON 消息体]
```

- 4 字节长度头：表示后续 JSON 消息体长度，采用大端序编码
- JSON 消息体：统一格式为 `type + data`

### 3.2 统一消息结构

```json
{
  "type": "MESSAGE_TYPE",
  "data": {
    "key": "value"
  }
}
```

- `type`：消息类型，相当于接口名
- `data`：该接口需要的业务字段

### 3.3 关键消息类型

客户端到服务端：

- `REGISTER`
- `LOGIN`
- `SEND_MESSAGE`
- `CREATE_GROUP`
- `JOIN_GROUP`
- `LEAVE_GROUP`
- `GET_GROUP_MEMBERS`
- `ADD_GROUP_MEMBER`
- `HEARTBEAT`
- `FILE_TRANSFER`
- `FILE_REQUEST`
- `RECALL_MESSAGE`

服务端到客户端：

- `REGISTER_RESPONSE`
- `LOGIN_RESPONSE`
- `MESSAGE`
- `USER_LIST`
- `GROUP_LIST`
- `USER_STATUS`
- `GROUP_MEMBERS`
- `GROUP_HISTORY_SYNC`
- `HEARTBEAT_RESPONSE`
- `FILE_DATA`
- `ERROR`

### 3.4 消息格式示例

登录请求：

```json
{
  "type": "LOGIN",
  "data": {
    "username": "user1",
    "password": "password123"
  }
}
```

聊天消息：

```json
{
  "type": "SEND_MESSAGE",
  "data": {
    "sender": "user1",
    "receiver": "group1",
    "content": "@AI 介绍一下你自己",
    "timestamp": 1715580000.0,
    "is_group": true
  }
}
```

心跳包：

```json
{
  "type": "HEARTBEAT",
  "data": {
    "username": "user1",
    "timestamp": 1715580000.0
  }
}
```

文件下载请求：

```json
{
  "type": "FILE_REQUEST",
  "data": {
    "username": "user2",
    "file_id": "user1_1715580000_notes.md",
    "offset": 4096
  }
}
```

## 4. 模块划分

### 4.1 客户端

- `client.py`
  - 客户端启动入口
  - 建立 socket 连接
  - 接收消息并分发
  - 心跳发送与自动重连
- `client_ui.py`
  - 登录界面、会话目录、聊天区、输入区、状态栏
- `client_history.py`
  - 结构化历史消息
  - 本地持久化
  - 未读状态管理
- `client_file_transfer.py`
  - 文件发送、文件通知、分块下载、断点续传

### 4.2 服务端

- `server.py`
  - 主服务端程序
  - 连接接入
  - 消息路由
  - 用户与群组管理
  - 心跳检测
- `llm_assistant.py`
  - 调用 GLM API 完成 AI 回复
- `content_analyzer.py`
  - 对消息做敏感词审核
- `file_transfer.py`
  - 文件块的创建、写入、读取和大小获取

## 5. AI 功能实现逻辑

### 5.1 触发方式

AI 功能只在群聊中启用。用户在群聊输入：

```text
@AI 问题内容
```

服务端收到群聊消息后，会先完成正常的群消息处理，再检查是否包含 `@AI`。

### 5.2 服务端处理流程

1. 客户端发送普通群聊消息 `SEND_MESSAGE`
2. 服务端先进行内容审核
3. 如果 `is_group=True` 且消息中包含 `@AI`
4. 服务端调用 `llm_assistant.py`
5. 将 AI 返回结果重新封装成一条普通群消息
6. 广播给群内所有成员，并写入群历史

### 5.3 为什么放在服务端

- 客户端无需保存 API Key
- 所有成员都能看到同一条 AI 回复
- AI 回复天然进入群消息历史
- 更方便在服务端统一做审核与日志管理

## 6. 文件传输设计

文件传输采用“两阶段”方式：

1. 上传阶段  
   发送方把文件按固定大小分块，上传到服务端缓存。
2. 下载阶段  
   接收方收到文件通知后，再按 `offset` 主动请求文件块。

关键机制：

- 每块大小：`FILE_CHUNK_SIZE = 4096`
- 二进制内容：用 Base64 编码后放进 JSON
- `offset`：块在整个文件中的起始位置
- `is_end`：是否最后一块

这样设计可以避免大文件一次性占满聊天通道，并支持断点续传。
